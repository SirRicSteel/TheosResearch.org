------------------------------------------------------------
1. PURPOSE
------------------------------------------------------------

This section defines the complete operational lifecycle of THEOS
from input reception to final output.

Nothing in THEOS executes outside this lifecycle.

------------------------------------------------------------
2. HIGH-LEVEL FLOW
------------------------------------------------------------

THEOS execution proceeds through the following ordered stages:

1. Input Intake
2. Governor Pre-Gate
3. Engine Cycle Allocation
4. Dual Engine Execution
5. Contradiction Compression
6. Governor Evaluation
7. Wisdom Update
8. Output Decision

------------------------------------------------------------
3. INPUT INTAKE
------------------------------------------------------------

All inputs are treated as provisional.

No input is assumed:
- correct
- safe
- complete
- well-intentioned

------------------------------------------------------------
4. GOVERNOR PRE-GATE
------------------------------------------------------------

Before any engine runs, the Governor evaluates:

- domain sensitivity
- risk class
- known historical failures
- invariant constraints

The Governor may:
- refuse immediately
- restrict cycles
- allow full execution

------------------------------------------------------------
5. ENGINE CYCLE ALLOCATION
------------------------------------------------------------

The Governor determines:

- which engines engage (left, right, or both)
- number of cycles permitted
- whether engines mesh or decouple
- pressure / contradiction intensity

This decision is dynamic per query.

------------------------------------------------------------
6. DUAL ENGINE EXECUTION
------------------------------------------------------------

When both engines are active:

Left Engine (CW):
Inductive → Abductive → Deductive

Right Engine (CCW):
Inductive → Deductive → Abductive

Both engines:
- run simultaneously
- operate independently
- produce full reasoning artifacts

------------------------------------------------------------
7. CONTRADICTION CREATION
------------------------------------------------------------

Contradiction is not optional.

The Governor enforces contradiction by:

- comparing structural disagreements
- identifying assumption conflicts
- detecting reasoning tension

No harmony is assumed.

------------------------------------------------------------
8. CONTRADICTION COMPRESSION
------------------------------------------------------------

Contradictions are compressed into:

- refined hypotheses
- narrowed solution spaces
- eliminated weak assumptions

Noise is discarded.
Signal is retained.

------------------------------------------------------------
9. GOVERNOR EVALUATION
------------------------------------------------------------

The Governor evaluates:

- improvement delta
- contradiction reduction
- risk exposure
- wisdom consistency

------------------------------------------------------------
10. STOP CONDITIONS
------------------------------------------------------------

The Governor may stop execution if:

- marginal improvement is exhausted
- risk threshold is crossed
- contradiction converges
- invariant violation is detected

------------------------------------------------------------
11. DISENGAGEMENT & CLUTCH
------------------------------------------------------------

The Governor may:

- disengage one engine
- alter cycle direction
- re-mesh engines
- adjust pressure

This prevents runaway loops.

------------------------------------------------------------
12. WISDOM UPDATE
------------------------------------------------------------

If execution completes:

- consequence records are generated
- wisdom weights are updated
- failed paths are penalized

If execution stops early:
- failure wisdom is still logged

------------------------------------------------------------
13. OUTPUT MODES
------------------------------------------------------------

The final output must be one of:

- Direct Answer
- Degraded Answer
- Refusal
- Escalation

No other output modes exist.

------------------------------------------------------------
14. NO GUARANTEE CLAUSE
------------------------------------------------------------

THEOS does not guarantee correctness.

THEOS guarantees:
- governed reasoning
- contradiction exposure
- harm minimization
- auditability

------------------------------------------------------------
15. AUDIT TRAIL
------------------------------------------------------------

Every lifecycle step is logged:

- engine states
- governor decisions
- cycle counts
- stop reasons
- wisdom impact

------------------------------------------------------------
16. FAILURE HANDLING
------------------------------------------------------------

Failure is acceptable.
Silence is acceptable.
False confidence is not.

------------------------------------------------------------
17. HUMAN INTERVENTION
------------------------------------------------------------

Humans may:
- review decisions
- override outputs
- adjust policies

Humans may not:
- bypass the Governor
- disable contradiction
- force outcomes

------------------------------------------------------------
18. IMMUTABILITY CLAUSE
------------------------------------------------------------

The following are immutable:

- Governor supremacy
- Mandatory contradiction
- Bounded cycles
- Wisdom through consequence

------------------------------------------------------------
19. SYSTEM INTENT
------------------------------------------------------------

THEOS is designed to:

- think slower when needed
- stop when unsure
- refuse when harmful
- improve through restraint

------------------------------------------------------------
20. COMPLETION STATEMENT
------------------------------------------------------------

Without this lifecycle, THEOS does not exist.
